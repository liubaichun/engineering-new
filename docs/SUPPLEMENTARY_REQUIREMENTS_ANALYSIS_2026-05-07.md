## 二、P0 紧急问题（阻断商业化）

### P0-1：多租户隔离未完成（影响所有模块数据）
**影响范围**：CRM(4表) + 采购(3表) + Tasks(11表) + Equipment(3表) + 通知/审计(3表) + User模型

**根因**：这些表没有 `company_id` 字段，API 返回数据无租户过滤

**修复方案**：
1. 为以下表添加 `company_id` 外键：
   - `crm_client`, `crm_contract`, `crm_supplier`, `crm_opportunity`
   - `crm_contract_changelog`, `crm_payment_plan`
   - `purchasing_purchaserequest`, `purchasing_purchaseorder`, `purchasing_purchasereceive`
   - `tasks_project`, `tasks_task`, `tasks_flowtemplate`, `tasks_taskstageinstance`, `tasks_taskflowinstance`
   - `equipment_equipment`, `equipment_equipmentusagelog`, `equipment_equipmentbomrelation`
   - `material_material`, `material_materialbom`, `material_materialbomnode`
   - `notifications_notificationlog`, `core_operationauditlog`, `core_loginlog`
   - `auth_user`（新增 `company_id` 字段）

2. 编写数据迁移：将历史数据回填 `company_id`（基于现有外键关系推导）
3. 后端 API 过滤：所有 QuerySet 加 `.filter(company_id=request.user.company_id)`
4. superuser bypass：管理员可看所有数据

**工作量**：中等（主要工作在前端字段映射+数据回填）

---

### P0-2：两套审批并存
**现状**：
- 旧审批：`approvals_approvalflow`（事件驱动）
- 新审批：`tasks_flowtemplate`（模板驱动）

**问题**：用户不知道用哪套，系统复杂度高

**建议方案**：
1. 保留 `tasks_flowtemplate` 作为标准审批模板
2. 旧审批 `ApprovalFlow` 保留用于报销/支出等财务场景（已经有 `ApprovalFlow` 实例在用）
3. 统一入口：所有审批都通过"审批管理"菜单发起，禁止从业务模块直接创建审批流

---

### P0-3：User模型无company_id
**影响**：无法实现真正的多租户数据隔离（用户可在 URL 层面绕过租户边界）

**修复**：为 `auth_user` 表添加 `company_id` 字段，迁移现有 superuser 到默认公司

## 三、P1 重要功能（影响核心业务链路）

### P1-1：工资管理链路断点
**链路**：员工管理 → 工资录入 → 银行流水 → 工资确认 → 发放

**当前断点**：
1. 员工管理菜单无入口（分散在工资管理页面里）
2. WageRecord 无 `detail` 路由（只有 list）
3. 银行流水导入：只能导入收支数据，无法自动关联到工资记录
4. 工资没有状态机：录入→确认→发放→已发

**补充需求**：
- `员工管理` 作为独立菜单项（已在菜单合并中添加）
- `银行流水导入` 增加"工资识别"规则：根据金额+日期+备注关键词，自动匹配未发放工资记录
- `工资记录` 增加状态：`draft → submitted → approved → paid`
- 工资明细页：显示每条工资对应的银行流水（是否已确认）

---

### P1-2：采购管理三合一页面
**当前**：采购申请/订单/入库是3个独立菜单

**建议**：合并为「采购管理」一个页面，顶部 Tab 切换
```
[采购申请] [采购订单] [采购入库]  ← Tab 切换
```

每个 Tab 内：
- 列表视图（表格）
- 支持快捷创建（申请→订单→入库的关联创建）
- 状态联动：订单被入库后，订单状态自动更新

**工作量**：低（已有3个完整页面，重构为Tab切换）

---

### P1-3：收支管理与发票联动
**当前**：收支记录和发票完全独立

**建议**：
- 收入/支出记录可关联发票号（`invoice_id` 外键）
- 发票管理页面增加"按收支记录开票"快捷入口
- 发票作废时自动冲销对应收支记录的"已开票金额"

---

### P1-4：设备BOM功能完善
**当前**：设备-配件关联已实现，但物料BOM树形结构不完整

**建议**：
- `MaterialBOM` 表已有（主件+版本），但 `MaterialBOMNode` 自引用树形节点功能未完成
- 物料详情页显示"BOM结构"Tab，支持：增加子件 / 设置数量 / 设置损耗率
- 设备详情页显示"关联物料BOM"Tab
- 采购申请支持"从BOM生成采购清单"功能

---

### P1-5：CRM多表company_id + 字段补全
**当前**：客户/合同/供应商/商机均无 `company_id`，且部分字段缺失

**建议**：
| 表 | 缺失字段 | 说明 |
|----|---------|------|
| Contract | `company_id` | 合同必须归属公司 |
| Contract | `total_amount` | 合同总金额 |
| Contract | `signed_date` | 签订日期 |
| ContractChangeLog | `company_id` | 变更记录同归属 |
| PaymentPlan | `company_id` | 付款计划同归属 |
| PaymentPlan | `paid_amount` | 已付款金额 |
| Supplier | `company_id` | 供应商归属公司 |
| Supplier | `contact_person` | 联系人 |
| Supplier | `payment_terms` | 付款条款 |

---

### P1-6：文件管理功能退化
**当前问题**：
- 文件上传后无分类管理
- 文件无法预览（图片/PDF）
- 批量下载缺失

**建议**：
- 增加文件分类管理（`FileCategory` 模型已存在但未使用）
- 文件列表增加「分类筛选」+「批量选择」+「ZIP打包下载」
- 图片/PDF 支持在线预览（使用 `xlsx/dist/jszip.min.js` 等前端库）

## 四、P2 优化需求（提升体验）

### P2-1：甘特图+任务看板优化
- 甘特图增加「指派给」筛选
- 任务看板增加「截止日期」筛选
- 甘特图导出为图片功能
- 项目详情页集成甘特图组件

### P2-2：审批规则自动化
- 系统参数中配置：支出≥1000 / 收入≥5000 → 自动触发审批流
- 预警中心：合同到期前30天自动提醒
- 工资超限预警：单人月工资>5万自动预警

### P2-3：数据导入导出
- 客户/供应商/员工 支持 Excel 批量导入
- 所有列表页支持 CSV 导出
- 银行流水支持 Excel/CSV 导入（已有框架，需完善）

### P2-4：通知渠道集成
- 飞书 Webhook 通知（已配置，但未实际推送）
- 邮件通知（需配置 SMTP）
- 审批结果推送（审批通过/拒绝后通知申请人）

### P2-5：操作审计日志
- 关键操作（删除/审批通过/工资录入）记录详细变更前后值
- 审计日志按模块+时间范围筛选
- 审计日志导出

## 六、技术债务

### 6.1 数据库
- 尽早从 SQLite 迁移到 PostgreSQL（多租户+并发性能）
- 所有表统一加 `created_at` / `updated_at`
- 所有业务表统一加 `company_id`

### 6.2 代码质量
- 统一 `User` 扩展方式：不用继承 `AbstractUser`，用 `Profile` 模式扩展
- 统一权限控制：所有 ViewSet 加 `CompanyOwnerFilter`
- API 文档：所有自定义字段加 `@extend_schema_field` 注解

### 6.3 前端
- 统一使用 `axios`（部分页面用 `fetch`，部分用 jQuery AJAX）
- 统一 ECharts 版本（目前两个页面用了不同 CDN）
- 统一日期选择器（避免混用 Bootstrap DateTime 和 HTML date input）
