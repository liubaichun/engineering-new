## 第二部分：🔴 P0 阻塞问题（必须立即修复）

### P0-1: 46 处 `objects.get()` 无异常处理

**风险：** 任何 `get()` 找不到记录都会抛出 `DoesNotExist`，直接返回500。用户看到白屏。

**涉及文件：**
```
apps/approvals/views.py         — 16处  get() without try
apps/crm/views.py               — 8处
apps/finance/views.py           — 7处
apps/tasks/views.py             — 6处
apps/core/views.py              — 4处
apps/channels/views.py          — 3处
apps/purchasing/views.py        — 2处
```

**修复方案：**
```python
# ❌ 当前写法（500风险）
invoice = Invoice.objects.get(id=invoice_id)

# ✅ 修复后
try:
    invoice = Invoice.objects.get(id=invoice_id)
except Invoice.DoesNotExist:
    return Response({'code': 2002, 'message': '发票不存在'}, status=404)
```

**工时估计：** 3-4小时（批量搜索替换＋手动审查每个上下文）

### P0-2: 93 处 `.save()` 无异常处理

**风险：** 数据库写入失败时不返回任何错误信息，用户以为保存成功了但数据没存进去。

**涉及文件（部分）：**
```
apps/approvals/views.py         — 16处
apps/finance/views.py           — 22处
apps/crm/views.py               — 13处
apps/tasks/views.py             — 11处
apps/core/views.py              — 7处
apps/channels/views.py          — 5处
apps/material/views.py          — 4处
apps/purchasing/views.py        — 3处
apps/equipment/views.py         — 2处
```

**修复方案：**
```python
# ❌ 当前写法
obj.save()

# ✅ 修复后
try:
    obj.save()
except Exception as e:
    logger.error(f"保存{obj.__class__.__name__}失败: {e}")
    return Response({'code': 5000, 'message': '保存失败'}, status=500)
```

**工时估计：** 2-3小时（批量修复，同类问题统一处理）

### P0-3: CSRF中间件未启用

**风险：** 整个系统没有CSRF防护，攻击者可构造恶意链接让管理员执行非授权操作。

**当前配置（config/settings.py）：**
```python
MIDDLEWARE = [
    # ... 没有 CsrfViewMiddleware ！
]
```

**修复方案：**
```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',  # ← 添加
    # ...
]
```

**注意：** 由于API端点使用Token认证（DRF），CSRF主要影响：
- 登录页面（表单登录需要CSRF token）
- 传统表单提交的视图（`templates/` 下的非API页面）
- 检查是否有视图使用了 `@csrf_exempt`，只在必要时保留

**工时估计：** 1小时（配置修改 + 前端模板加 `{% csrf_token %}`）

## 第三部分：🟠 P1 紧急问题（本周内修复）

### P1-1: raw SQL 注入风险（3处）

**文件：** `apps/core/permissions.py` 第101、343、374行

```python
# 当前写法（%d格式化，有注入风险）
cursor.execute("SELECT ... WHERE id = %d" % some_id)
```

**修复方案：** 替换为参数化查询：
```python
cursor.execute("SELECT ... WHERE id = %s", [some_id])
```

**工时：** 30分钟

### P1-2: 安全HTTP头配置（4项）

| 配置项 | 当前值 | 应设为 | 影响 |
|-------|--------|-------|------|
| `SECURE_HSTS_SECONDS` | 未设置 | `31536000` | 强制HTTPS |
| `SECURE_SSL_REDIRECT` | False | True | HTTP→HTTPS重定向 |
| `SESSION_COOKIE_SECURE` | False | True | Session Cookie仅HTTPS |
| `X_FRAME_OPTIONS` | 未设置 | `'DENY'` | 防点击劫持 |

**修复方案：** 在 `config/settings.py`中添加：
```python
SECURE_HSTS_SECONDS = 31536000
SECURE_SSL_REDIRECT = True  # 注意：本地开发时需注释掉
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
X_FRAME_OPTIONS = 'DENY'
```

**工时：** 15分钟

### P1-3: 58处 bare `except Exception` 无日志

**风险：** 异常被吞掉，排查问题时完全不知道发生了什么。

**重灾区：**
```
apps/finance/bank_adapters.py    — 21处  ❌
apps/finance/bank_import_views.py — 7处
apps/repair/views.py               — 6处
```

**修复方案：** 所有 `except` 块中至少加一行日志：
```python
# ❌ 当前
except Exception:
    pass  # 或 return 某个值

# ✅ 修复后
except Exception as e:
    logger.exception(f"银行适配器异常: {e}")
    raise  # 或返回错误响应
```

**工时：** 2小时

### P1-4: `on_delete=CASCADE` 关键业务模型

**风险：** 删一个员工，可能连带删掉工资记录、社保记录、审批流——灾难级联删除。

**需要改为PROTECT的模型：**
```python
# finance/models.py (15处CASCADE风险)
SocialRecord.employee          → PROTECT  # 删员工→社保记录全没了
Budget.company                 → PROTECT  # 删公司→预算全没了
WageRecord.employee_company    → PROTECT  # 删任职→工资记录没了

# core/models.py (10处)
UserCompanyRole.user           → PROTECT  # 删用户→角色关联没了
Notification.user              → CASCADE合理（通知是用户的附属）
```

**原则：** 业务数据 `on_delete=PROTECT`，附属数据 `on_delete=CASCADE`

**工时：** 3小时（涉及migration需要处理已有数据）

### P1-5: 缺失权限校验（API端点暴露）

**发现：** 仅7处明确使用了 `@login_required` / `has_perm` / `@permission_required`。大量视图依赖DRF默认权限类，但部分视图没有显式声明 `permission_classes`。

**需要检查的视图：**
- `apps/equipment/views.py` - 8个视图集，权限类不明确
- `apps/repair/views.py` - 6个视图，缺少权限声明
- `apps/files/views.py` - 4个视图，缺少权限声明
- `apps/purchasing/views.py` - 部分视图缺少权限

**修复方案：** 逐视图检查并补全 `permission_classes = [IsAuthenticated]` 或自定义权限

**工时：** 4小时

### P1-6: `print()` 替代日志（生产代码）

**文件：**

## 第四部分：🟡 P2 重要问题（本月内修复）

### P2-1: 80+处 `.all()` 无过滤

**风险：** 多租户场景下，`Model.objects.all()` 如果不带 `company_id`/`tenant_id` 过滤，可能暴露其他公司的数据。

**涉及文件（部分）：**
```
apps/tasks/views.py      — 12处
apps/core/views.py       — 10处
apps/crm/views.py        — 8处
apps/purchasing/views.py — 6处
```

**修复方案：** 结合多租户中间件，确保所有 `get_queryset()` 自动注入 `company_id` 过滤：
```python
def get_queryset(self):
    return self.queryset.filter(company=self.request.company)
```

**工时：** 2天（需要逐视图确认+配合多租户改造）

### P2-2: `filter()` 缺少 `order_by()`（6处）

**风险：** 分页结果顺序不固定，用户刷新页面看到数据顺序变化。

**涉及：**

apps/finance/reports_v2.py    — 过多处 filter without order

### P2-5: 序列化器缺少类型注解（230+处）

**现状：** drf_spectacular 报W001警告230+次，所有 `SerializerMethodField` 的 `get_xxx` 方法缺少返回类型注解。

**影响：** API文档（Swagger）无法正确显示字段类型和示例值。

**修复方案：**
```python
# ❌ 当前
def get_status_display(self, obj):
    return obj.get_status_display()

# ✅ 修复后
def get_status_display(self, obj) -> str:
    return obj.get_status_display()
```

**工时：** 2小时（批量正则替换+手动补充复杂类型）

### P2-6: Setting配置分散

**现状：** 部分配置散落在 `config/settings.py` 之外，如 `apps/core/apps.py` 中做了数据库查询（导致 `AppConfig.ready()` 警告）。

**修复方案：** 将延迟初始化移出 `AppConfig.ready()`，改用 lazy 加载。

**工时：** 1小时

### P2-7: 临时文件清理

**现状：** 根目录下有大量 `tmp_*.py` 文件（`tmp_audit.py`, `tmp_sr_bug.py`, `tmp_users.py` 等17+个），`static/` 下有非静态文件。

**修复方案：** 删除所有临时文件，`static/` 清理

**工时：** 15分钟

### P2-8: 迁移文件积累（172个）

**现状：** 172个迁移文件，其中部分（如 `0009_remove_approvalflow_company`→`0010_add_company_id`）是来回修改的产物。

**建议：** Phase 0 结束后做一次 `squashmigrations`，合并到1-2个迁移文件。
