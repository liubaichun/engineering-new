## 二、核心问题：分类体系缺失

### 当前问题根因

银行流水导入时，`expense_category` 字段被填成了**银行给的交易类型描述**（"对公转账正常提出""企业银行各项费用"等），而不是**财务科目分类**。这就是为什么：

- 预算执行表用 `expense_category__icontains` 关键词匹配永远匹配不对
- 税费汇总要从描述中解析税单号才能知道是什么税

### 实际数据验证

**Income 的真实业务含义分布（219条/¥21,548,869）：**

| 银行交易类型 | 条数 | 金额占比 | 对应财务科目 |
|------------|------|---------|------------|
| 提回对公户收款 + IBPS对公提回贷记 | 159(73%) | ~74% | **主营业务收入**（客户付款） |
| 账户结息 | 38(17%) | ~1% | **财务费用-利息收入** |
| **内部公司转账** | 18(8%) | **¥1,680,107 / 7.8%** | **→ 应排除/单独标注** |
| 其他 | 4(2%) | ~0.1% | 往来款 |

**Expense 的真实业务含义分布（975条/¥20,508,102）：**

| 银行交易类型 | 条数 | 金额占比 | 对应财务科目 |
|------------|------|---------|------------|
| 对公转账正常提出 | 287(29%) | ~30% | **采购成本/项目支出**（需看摘要细分） |
| 税款 | 269(28%) | ~28% | **应交税费**（实际缴税） |
| 企业银行各项费用 | 120(12%) | <1% | **财务费用-手续费** |
| **内部公司转账** | 18(2%) | **¥1,720,003 / 8.4%** | **→ 应排除** |
| 代发工资 | 57(6%) | ~6% | **应付职工薪酬** |
| 服务费 | 30(3%) | ~3% | **管理费用** |
| 其他费用+备用金+房费+报销等 | 98(10%) | ~10% | 按摘要细分 |
| 代发手续费+转账收费等 | 22(2%) | <1% | **财务费用** |

## 四、详细转型规划

### Phase 1: 分类体系建设（当前缺失层）

#### 1.1 收入自动分类规则

基于 `Income.transaction_type` + `Income.customer` 字段：

```python
INCOME_CLASSIFICATION_RULES = [
    # (条件, 科目编码, 科目名称)
    ('transaction_type 包含 "结息"',          '04', '利息收入'),
    ('customer 在内部公司名单内',               '99', '内部转账'),  # 需排除
    ('transaction_type in ("提回对公户收款","IBPS对公提回贷记")', '01', '主营业务收入'),
    ('transaction_type = "还款"',              '03', '往来款回收'),
    ('default',                               '02', '其他业务收入'),
]
```

**当前数据覆盖度：** Income 219条中，按此规则可自动分类 215条（98%），仅4条需人工确认。

#### 1.2 支出自动分类规则

基于 `Expense.transaction_type` + `Expense.supplier` + `Expense.description`：

```python
EXPENSE_CLASSIFICATION_RULES = [
    ('transaction_type in ("代发工资","代发奖金")',    '21', '工资薪酬'),
    ('transaction_type in ("税款","代理国库税收收缴")',  '31', '税费'),
    ('transaction_type in ("企业银行各项费用","转账收费","凭证出售收费","代发手续费")', '15', '财务费用-手续费'),
    ('supplier = "深圳市住房公积金管理中心"',            '22', '公积金'),
    ('supplier 在内部公司名单内',                       '99', '内部转账'),
    ('supplier in ("刘柏春","沈容","王铁印","韦东荣"...)','21', '员工报销/薪酬'),
    ('supplier = "暂收款" and transaction_type = "税款"', '31', '税费'),
    ('transaction_type = "服务费"',                       '12', '管理费用-服务费'),
    ('transaction_type = "房费"',                          '12', '管理费用-房租'),
    ('default',                                           '11', '营业成本/项目支出'),
]
```

**当前数据覆盖度：** Expense 975条中，按此规则可自动分类 856条（88%），剩余119条"对公转账正常提出"需要按 `summary` 文本进一步细分。

**方案选择：** 推荐不加新字段/新表，用**Python分类函数**在报表查询时动态计算，理由：

- 分类规则可灵活调整（无需migration）
- 性能足够（219+975条数据，计算在毫秒级）

### Phase 2: 报表重构

#### 2.1 改名（立即做，不影响数据）

| 当前名 | 建议名 | 原因 |
|-------|-------|------|
| 资产负债表 | → **收支汇总表** | 不是资产=负债+权益，只是收入减支出 |
| 现金流量表 | → **银行余额变动表** | 不是按经营/投资/筹资分类 |

#### 2.2 利润表（新增，P0）

基于分类体系的收入-支出按科目汇总：

```
营业收入: 主营业务收入 + 其他业务收入          ¥X
减: 营业成本                                  ¥X
────────────────────────────────────────
毛利                                          ¥X
减: 管理费用                                  ¥X
减: 销售费用                                  ¥X
减: 研发费用                                  ¥X
减: 财务费用                                  ¥X
加: 利息收入                                  ¥X
────────────────────────────────────────
营业利润                                      ¥X
加: 营业外收入                                ¥X
减: 营业外支出                                ¥X
────────────────────────────────────────
利润总额                                      ¥X
减: 所得税费用                                ¥X
────────────────────────────────────────
净利润                                        ¥X
```

#### 2.3 收支汇总表（改名的balance_sheet）

保留当前收入-支出的计算逻辑，但：
- **排除内部转账**（7.8%的干扰数据）
- 增加多公司合并视图（集团总收支）

#### 2.4 银行余额变动表（改名的cash_flow）

保持当前逻辑不变，这是准确的数据。

#### 2.5 税费汇总修复（P1）

- 统一社保计算：全部走 SocialRecord
- 增值税：显示"销项税 - 进项税 = 应交增值税"
- 个税：从 WageRecord 读取（比从银行流水解析更准确）

### Phase 3: 内部转账处理

**当前影响：** 内部转账占收入7.8%、占支出8.4%，如果不排除会严重歪曲单个公司的利润。

**处理策略：**
1. **集团合并视图** — 内部转账自动抵消
2. **单一公司视图** — 内部转账标注为"内部往来"不记入收入/支出
3. **全部公司视图（当前）** — 排除内部转账后展示集团真实对外收支

```
集团真实对外收入 = ¥21,548,869 - ¥1,680,107 = ¥19,868,762
集团真实对外支出 = ¥20,508,102 - ¥1,720,003 = ¥18,788,099
集团真实利润 = ¥19,868,762 - ¥18,788,099 = ¥1,080,663
                                ↑ 比未排除时多了 ¥39,896
```

### Phase 4: 数据补充建议

| 数据缺口 | 影响 | 补充方式 |
|---------|------|---------|
| 社保仅4条(1月) | 工资汇总/税费汇总社保部分不完整 | 需持续导入社保数据 |
| Expense的`expense_category`字段填错 | 预算执行表不准 | 分类后自动修复 |
| "对公转账正常提出"287条未细分 | 营业成本/管理费用无法拆分 | 需按摘要关键词二次分类 |
| 发票的 due_date 完整度 | 应收账龄准确性 | 导入时补全到期日 |

## 六、关键决策点

1. **分类规则放在哪？**
   - ✅ 推荐：独立 Python 函数模块 `classification_rules.py`，报表查询时动态调用
   - ❌ 不推荐：加数据库字段（需要 migration、影响导入逻辑、规则调整成本高）

2. **内部转账如何处理？**
   - ✅ 推荐：分类时标记为 `99-内部转账`，各报表按视图决定是否排除
   - 单一公司视图：标注为内部往来，不计入收入/支出
   - 集团合并视图：自动抵消

3. **利润表数据精细度？**
   - 当前数据已能支撑到"毛利→营业利润"级别
   - 更细的"主营业务收入-按产品拆分"需要补充项目关联数据
