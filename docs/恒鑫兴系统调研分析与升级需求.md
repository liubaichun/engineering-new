# 恒鑫兴系统调研分析与GREEN ERP升级需求方案

> 调研日期：2026-05-28
> 调研来源：https://hxx.ningmaoyun.com（狞猫云企业运营服务云平台 V3.64 · 恒鑫兴租户）
> 声明：本报告基于公开系统界面观察，提炼设计与业务逻辑，非代码抄袭。所有建议均结合GREEN ERP现状做定制适配。

---

## 第一部分：多租户架构深度分析

### 1.1 总体架构模式

**结论：子域名路由 + JWT租户ID隔离 + 共享数据库**

```
┌─────────────────────────────────────────────────────────────────┐
│                    ningmaoyun.com (SaaS平台)                      │
│                                                                   │
│  hxx.ningmaoyun.com  │  abc.ningmaoyun.com  │  xyz.ningmaoyun.com │
│  (恒鑫兴·租户ID:2)   │  (其他租户·ID:3)     │  (其他租户·ID:4)    │
├──────────────────────┼──────────────────────┼─────────────────────┤
│                      │      共享数据库                           │
│                      │  ┌────────────────────────────────────┐  │
│                      │  │ 所有表都有 tenant_id 字段          │  │
│                      │  │ 每个请求JWT带 TENANT_ID            │  │
│                      │  │ SQL WHERE tenant_id = ? 自动拦截   │  │
│                      │  └────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 关键技术细节

| 层面 | 实现方式 | 证据 |
|------|----------|------|
| **租户路由** | Nginx反向代理，按子域名 `hxx.ningmaoyun.com` 分发到对应租户上下文 | URL: `https://hxx.ningmaoyun.com/login` |
| **身份认证** | JWT Token存在Cookie（`Admin-Token`），每次请求携带 | Cookie中的JWT payload 含 `TENANT_ID: "2"` |
| **租户标识** | JWT extendedData 含两个字段：`TENANT_ID` + `HOST_TENANT_ID`（两字段相同，为同一租户；不同则可能是跨租户操作） | Token解密结果 |
| **数据隔离** | 所有数据库表 `WHERE tenant_id = ?` | 无独立DB，共享数据库模式 |
| **发行方** | `"iss": "caracal"` — 平台代号 | JWT头部 |
| **用户ID格式** | `1000000211599369259` — 全局统一ID空间（跨租户不重复） | JWT authUser.id |
| **租户配置** | 每个租户在系统设置→企业信息中独立配置Logo、名称、地址、参数 | 系统设置→参数设置页面 |

### 1.3 登录流程推演

```
用户访问 hxx.ningmaoyun.com
    ↓
DNS解析 → Nginx识别子域名 "hxx"
    ↓
反向代理到应用服务，携带租户标识
    ↓
服务端根据 hxx 查出 tenant_id = 2
    ↓
返回登录页，页面标题显示 "恒鑫兴运营服务云平台"
    ↓
用户输入账号密码
    ↓
服务端校验：用户属于tenant_id=2的租户
    ↓
生成JWT Token，payload含 TENANT_ID=2
    ↓
前端携带Token请求API，每次请求服务端中间件拦截
    自动在SQL中加 WHERE tenant_id = 2
```

### 1.4 可借鉴的优秀设计

| 序号 | 设计点 | 借鉴价值 | 我们的改进应用 |
|------|--------|----------|---------------|
| 1 | **子域名租户路由** | 高 | 我们的系统目前用公司切换（session），可考虑增加子域名直入（如 `bc.erp.com` 直接进入百川） |
| 2 | **JWT携带TENANT_ID** | 高 | 我们已有类似设计，可参考其 `extendedData` 结构扩展 |
| 3 | **全局统一用户ID空间** | 中 | 用户ID跨租户不重复，便于平台级审计和管理 |
| 4 | **自动SQL拦截 `WHERE tenant_id=?`** | 高 | Django middleware 自动注入 company_id 过滤，我们已有但可更严谨 |
| 5 | **三级数据权限**（本人/部门/全部） | 高 | 我们目前只有角色权限，缺少行级数据隔离 |
| 6 | **租户级参数独立配置** | 中 | 各公司独立配置Logo、精度、编码规则等 |
| 7 | **编码规则模板化** | 高 | `{T:前缀}{D:日期}{SEQ:长度:起始}` 格式可配，且每个租户独立序列 |

### 1.5 不足之处与风险

| 问题 | 风险 | 我们的对策 |
|------|------|-----------|
| **子域名硬编码**：租户代码写在代码/配置里，新增租户需改Nginx配置 | 扩容不灵活 | 采用**数据库路由表**：一个通配符域名 `*.erp.com`，路由表动态匹配租户 |
| **共享DB无分库**：所有租户在同一张表，数据量大时查询性能下降 | 性能瓶颈 | 我们目前的公司数量少（4家），短期共享DB可行；长期可设计**按租户分库**策略 |
| **TENANT_ID暴露在前端JWT**：理论上可篡改 | 越权风险 | JWT加签名校验 + 服务端二次验证租户与用户匹配 |
| **仅字段级隔离**：没有独立的Schema或数据库 | 数据恢复/备份粒度粗 | 可设计每个租户独立Schema（PostgreSQL Schema）或独立数据库 |
| **无可见的多租户管理后台**：没有看到超管界面 | 运营成本高 | 必须提供**平台管理端**，能看到所有租户、用量、订单 |
| **租户注册/试用流程缺失**：登录页无"注册"入口 | 获客成本高 | 设计自助注册→自动创建租户→初始化数据流程 |

## 第二部分：按您的要求——个性化方案（避免"抄袭感"）

> 核心原则：**学思想，不抄界面；结合自身业务特色做差异化**

### 2.1 我们的差异化定位

| 维度 | 恒鑫兴/狞猫云 | GREEN ERP（我们的定位） |
|------|--------------|----------------------|
| **行业** | 通用制造业（设备/配件） | 软件产品认证导向 + 财务合规 |
| **核心** | 进销存+财务一体化 | 项目管理+认证+财务合规 |
| **特色** | 分销二维码、多维度销售分析 | **工资/社保/预算/审批**（恒鑫兴没有） |
| **架构** | 传统单体SPA | 多Agent协同+自动化任务 |
| **UI风格** | 传统企业级UI | 简洁现代风 |

### 2.2 差异化改造方案（避免雷同）

#### 🎯 销售管理——不走"传统订单"路线

恒鑫兴的销售订单是**传统进销存模式**：
```
报价→订单→发货→出库→收款
```

我们可改为**项目驱动型销售**：
```
客户商机 → 项目立项 → 报价/合同 → 项目交付 → 里程碑付款
                  ↓
             关联研发任务、采购需求、生产计划
```

差异点：
- 不叫"销售订单"，叫"**销售项目**"或"**合同工单**"
- 每一笔销售关联一个项目生命周期
- 收款按**项目里程碑**触发，不按发货
- 已存在的**审批流**直接关联到销售合同审批

#### 🏭 库存/物料——做"轻量级物料协同"

恒鑫兴有完整的进销存（出入库、调拨、盘点、1690种物料），我们不需要照搬全做。

建议：
- 不做"仓库管理"这个传统概念
- 改成"**物料流转看板**"
- 聚焦：**物料从采购→到货→领用→消耗**这个闭环
- 增加我们独有的：**物料与项目关联**（这个项目用了哪些物料？）
- 增加**物料预算**（预算管理我们已经有了，可关联物料领用）

#### 📊 报表——做"管理驾驶舱"而不是"统计表"

恒鑫兴有11种销售统计报表，我们不做一一对应：
- 做**一个综合管理驾驶舱**，用图形化展示
- 把我们的**预算、工资、社保、项目进度**等独有数据融入
- 突出**我们有的他们没有的**：项目预算执行率、工资成本占比、社保合规率

#### 🏢 客户管理——做"客户360°"

恒鑫兴有客户跟进、拜访、分销，我们：
- 不做分销二维码（对我们行业不适用）
- 做**客户360°视图**：合同→项目→应收款→发票→历史往来
- 增加**认证相关**：客户证书管理、合同到期提醒
- 结合**审批流**：客户信用审批、合同审批

### 2.3 界面风格差异化

| 方面 | 恒鑫兴 | GREEN ERP（建议） |
|------|--------|-----------------|
| 配色 | 蓝白传统企业风 | 保持现有风格 |
| 表格 | 传统表格+分页 | 可考虑卡片式+列表切换 |
| 仪表盘 | 销售简报+应收应付+业绩排行 | 加入**预算执行率、工资社保占比、项目进度、认证到期提醒** |
| 操作 | 每行多个操作按钮 | 采用**"更多"下拉菜单**收敛操作项 |

## 第三部分：升级需求（更新版·融入多租户分析）

### 3.1 多租户架构优化（新增）

| 优先级 | 改进项 | 说明 |
|--------|--------|------|
| P0 | **子域名租户路由** | 支持 `bc.erp.com` → 百川, `ljn.erp.com` → 绿聚能，无需手动切换公司 |
| P0 | **JWT租户校验增强** | 在中间件层加固 tenant_id 与用户的匹配校验 |
| P1 | **租户级配置独立** | Logo、名称、编码规则、精度等每个公司独立配置 |
| P1 | **数据权限三级体系** | 本人数据/部门数据/全部数据（目前只有角色菜单权限） |
| P2 | **编码规则模板化** | `{T:前缀}{D:日期}{SEQ:长度:起始}` 每个公司独立可配 |
| P2 | **平台管理端** | 超管后台：查看所有公司、用户数、数据量、系统健康 |

### 3.2 五大核心升级模块（P0→P2）

#### P0：库存与物料流转
- ✅ 多仓库（成品/配件/原料）
- ✅ 出入库流水
- ✅ 物料与项目关联（我们的特色）
- ✅ 物料预算（与现有预算模块联动）
- ❌ 不做：调拨单、组装拆装（初期不需要）

#### P0：销售合同与项目管理
- ✅ 销售合同/工单（非传统订单）
- ✅ 合同与项目关联
- ✅ 里程碑付款计划
- ✅ 合同审批（复用现有审批流）
- ✅ 开票申请（与现有发票模块联动）

#### P0：应收/应付款管理
- ✅ 合同自动生成应收款
- ✅ 收款单核销
- ✅ 到账提醒
- ✅ 账龄分析

#### P1：客户360°视图
- ✅ 客户→合同→项目→应收→发票→历史
- ✅ 客户跟进记录（轻量版）
- ✅ 认证到期提醒（我们的特色）
- ❌ 不做：分销、积分、拜访计划

#### P1：管理驾驶舱
- ✅ 销售简报
- ✅ 应收/应付图表
- ✅ **预算执行率**（我们独有）
- ✅ **工资社保成本占比**（我们独有）
- ✅ **项目进度概览**（我们独有）

### 3.3 保持不变（不盲目跟风）

以下功能恒鑫兴有，但我们**不需要**：

| 功能 | 原因 |
|------|------|
| 分销二维码 | 软件行业不适用 |
| 委外加工 | 我们没有生产制造 |
| 组装/拆装（BOM） | 初期不需要 |
| 对账单（给客户的） | 有发票就够了 |
| 11种销售统计报表 | 用管理驾驶舱替代 |
| 商品动销率 | 非零售行业 |
| 打印模板管理 | 后期再考虑 |

## 第四部分：实施路线图（按周划分）

| 阶段 | 周期 | 内容 |
|------|------|------|
| **Phase 1** | 第1-2周 | 多租户加固（子域名路由+JWT校验+租户配置独立） |
| **Phase 2** | 第3-5周 | 库存与物料流转（多仓库+出入库+物料-项目关联） |
| **Phase 3** | 第6-8周 | 销售合同与项目管理（合同工单+里程碑付款+审批） |
| **Phase 4** | 第9-10周 | 应收/应付款管理（自动生成+核销+账龄） |
| **Phase 5** | 第11-12周 | 客户360° + 管理驾驶舱 |
| **Phase 6** | 持续 | 编码规则模板化、数据权限体系、平台管理端 |
