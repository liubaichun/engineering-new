# GREEN ERP 系统升级方向需求分析报告 v3.0

> 文档日期：2026-05-29 | 版本：v3.0
> 文档说明：本版基于v2.0重构，核心调整为 **"通用平台 + 行业插件包"** 模式。不再做单一垂直系统，而是建设可承载多种企业类型的SaaS平台底座，行业特色功能以独立插件包形式按需加载。
> 定位原则：**不做大而全的金蝶用友，不做窄而深的单一垂直，做"平台核心通用+行业包可插拔"的灵活SaaS。**

## 第二部分：升级战略——"平台+行业包"模式

### 2.1 为什么要改策略

```
之前（v2.0思维）：
  "我们不做制造业，只做软件认证+财务合规垂直系统"
      ↓ 局限：
      · 市场天花板低
      · 恒鑫兴等竞品足够好
      · 我们已经在给"有制造属性的企业"服务

现在（v3.0思维）：
  "我们做通用SaaS平台底座 + 可选行业插件包"
      ↓ 优势：
      · 核心平台一套代码覆盖所有企业
      · 行业包按需加载，各自专注
      · 可同时服务认证行业、制造行业、甚至混合型企业
      · 不跟金蝶用友正面竞争，用差异化取胜
```

### 2.3 行业包定义

#### 认证行业包（Package A）—— 我们的主战场

> 定位：面向软件/电子产品认证、检测、合规类企业

| 模块 | 说明 | 对应v2.0升级方案 |
|------|------|----------------|
| 认证证书全生命周期 | 证书类型/编号/发证/到期/续期 | ClientCertification |
| 项目驱动型销售 | 商机→报价→合同→里程碑交付→收款 | SalesOpportunity + SalesContract + ReceivablePlan |
| 客户360°视图 | 合同→项目→应收→发票→认证→跟进 | 客户360°页面 |
| 审计追踪 | 认证过程全链路追溯 | AuditLog |
| 行业报表 | 认证到期日历/证书概览/项目成本 | Dashboard |

#### 制造行业包（Package B）—— 第二增长曲线

> 定位：面向中小型制造企业（进销存+BOM+委外+传统销售）
> **参考来源：** 基于对恒鑫兴（狞猫云V3.64）制造模块的分析。参考其功能设计但做简化，聚焦中小企业核心需求。

| 模块 | 子功能 | 说明 | 优先级 |
|------|--------|------|--------|
| **多仓库管理** | 仓库定义+库存流水+锁定/可用库存 | 支持多个仓库，库存按仓库隔离；增加锁定库存（销售预留）+可用库存逻辑 | P1 |
| **入库管理** | 采购入库+成品入库+退货入库+盘盈 | 关联采购订单；质检标记(合格/不合格+待检) | P1 |
| **出库管理** | 销售出库+领料出库+盘亏+退货出库 | 关联销售订单/生产工单；支持锁定库存自动释放 | P1 |
| **库存盘点** | 盘点计划→录入→差异审核→调整 | 支持全盘+抽盘；差异自动生成盘盈/盘亏单 | P1 |
| **调拨管理** | 仓库间调拨+在途库存 | 调拨申请→审批→发货→收货 四步流程 | P1 |
| **库存预警** | 最低库存+呆滞分析(90天未动) | 触发自动通知（企微/飞书/钉钉） | P1 |
| **历史价格管理** | 采购价格+销售价格 | 按物料+供应商记录历史采购价；按物料+客户记录销售价 | P1 |
| **传统销售订单** | 报价单→销售订单→发货→出库→应收 | 传统模式（非项目驱动），与核心平台应收模块对接 | P2 |
| **BOM+生产工单** | MBOM→生产计划→领料→加工→完工入库 | 基于现有MaterialBOM扩展；参考恒鑫兴的组装/拆装功能 | P2 |
| **委外加工** | 委外需求→委外订单→发外→加工费→回库 | 委外库存独立管理；加工费结算与应付对接 | P2 |
| **销售费用** | 销售过程中产生的费用管理 | 运费/安装费/调试费等 | P2 |
| **制造驾驶舱** | 库存周转率+生产进度+订单交付率+委外统计 | 制造企业特有的KPI指标 | P2 |
| **成本核算** | 料(物料)+工(工资)+费(费用)归集 | 按工单/按批次核算产品成本 | P3 |
| **销售对账单** | 给客户的对账报表 | 恒鑫兴有此功能，制造企业刚需 | P3 |
| **打印模板** | 单据打印模板自定义 | 恒鑫兴有此功能 | P3 |

> **关键原则：制造包不做深度，做到"够用好用"即可，不与金蝶用友拼深度。制造包的目标是覆盖恒鑫兴约80%的核心功能，但做得更轻、更易用。**

### 2.4 行业包如何工作

```python
# 租户配置——启用的行业包
class TenantPackage(models.Model):
    """租户-行业包关联"""
    PACKAGE_CHOICES = [
        ('core', '平台核心（必选）'),
        ('certification', '认证行业包'),
        ('manufacturing', '制造行业包'),
    ]
    
    tenant = models.ForeignKey(Tenant, on_delete=models.CASCADE)
    package = models.CharField('行业包', max_length=30, choices=PACKAGE_CHOICES)
    is_active = models.BooleanField('启用', default=True)
    activated_at = models.DateTimeField('启用时间', auto_now_add=True)
    
    class Meta:
        unique_together = [('tenant', 'package')]

# 菜单根据行业包动态加载
def get_menu_for_tenant(tenant):
    packages = TenantPackage.objects.filter(tenant=tenant, is_active=True)
    menu = []
    if any(p.package == 'certification' for p in packages):
        menu += CERTIFICATION_MENU_ITEMS
    if any(p.package == 'manufacturing' for p in packages):
        menu += MANUFACTURING_MENU_ITEMS
    # 核心菜单始终加载
    return CORE_MENU_ITEMS + menu
```

**用户体验效果：**
- 认证企业登录 → 看到：核心功能 + 认证证书管理 + 项目驱动销售
- 制造企业登录 → 看到：核心功能 + 进销存 + BOM + 委外加工
- 混合企业登录 → 看到：全部功能
- 对用户来说，感觉这就是专门为他们行业定制的系统

### 2.5 差异化竞争定位

| 对比对象 | 我们的定位 | 卖点 |
|---------|-----------|------|
| vs **金蝶/用友** | 轻量级SaaS，不拼功能深度 | 价格低（1/5~1/10）、上手快（3天vs3月）、工资社保审批做得比他们深 |
| vs **恒鑫兴/狞猫云** | 平台+行业包更灵活 | 核心功能覆盖恒鑫兴80% + 独有的工资社保预算审批；行业包按需加载不冗余 |
| vs **钉钉/企微生态** | 深度融合工资社保审批 | 不是轻表单，是有深度的业务系统（个税计算+社保分险种+7级累进税率） |
| vs **垂直SaaS** | 平台底座+多行业覆盖 | 企业成长不用换系统，加行业包即可 |

### 3.2 Phase 0：核心平台加固（第1-4周）

> 目标：补齐所有模块的断点，让核心平台真正"通用可用"

| W2 | **库存流水化** | StockTransaction模型（入库/出库/调拨/盘点）；废除直接修改Material.stock | 3天 | 所有库存变动有流水可追溯 |

## 第四部分：标准与规范

### 4.1 行业包开发标准

```yaml
# 行业包开发规范
行业包必须遵守:
  - 包内模块不能修改核心平台的代码（开闭原则）
  - 包内模块可以扩展核心平台的功能（通过信号/钩子）
  - 所有菜单、权限、路由自带行业包前缀（如 cert_xxx, mfg_xxx）
  - 数据模型必须继承 TenantAwareModel（自动隔离）
  - 行业包之间不能有耦合（认证包和制造包互不依赖）

目录结构:
  apps/
    core/              # 平台核心（不变）
    certification/     # 认证行业包
      models.py
      views.py
      serializers.py
      urls.py
      menus.py         # 菜单定义
      permissions.py   # 行业包权限
    manufacturing/     # 制造行业包
      models.py
      views.py
      serializers.py
      urls.py
      menus.py
      permissions.py
```

## 第五部分：实施保障

### 5.1 质量管理

```
每个阶段的发布标准：
  □ 功能完整：所有P0功能可操作
  □ 性能达标：页面加载 < 2秒，API响应 < 200ms
  □ 隔离验证：行业包A的功能不会污染行业包B的界面
  □ 回滚方案：行业包可一键停用
  □ 文档就绪：CHANGELOG + 使用说明 + 配置指南
```

### 5.2 风险控制

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| 制造包做得太重（功能膨胀） | 中 | 高 | 严格控制在P3范围内，超过的放到P4 |
| 核心平台改得太多影响现有用户 | 高 | 高 | 每个改动必须有兼容旧数据的migration |
| 行业包之间的冲突（菜单/权限/路由） | 中 | 中 | 强制命名前缀隔离 |
| 开发资源不足 | 中 | 高 | Phase 0-1优先，Phase 2-3可推迟 |

### 5.3 迭代节奏

```yaml
每个行业包的生命周期:
  1. 设计评审（1周）: 原型+数据模型+交互设计
  2. 核心开发（2-4周）: 模型→API→页面→测试
  3. 内部验收（1周）: 端到端全流程测试
  4. 灰度发布（1周）: 选1-2个友好客户试用
  5. 正式发布: 全量推+文档更新
```

## 第六部分：我们不做清单（坚定边界）

| 功能 | 原因 | 优先级 |
|------|------|--------|
| 详细生产排程(MES) | 太深，金蝶用友强项 | ❌ 不做 |

| 电商对接（淘宝/京东） | 不是我们的客户场景 | ❌ 不做 |
| HR全模块（考勤/绩效/招聘） | 只做工资社保审批 | ❌ 不做 |
| 项目管理甘特图 | 用任务替代，不做专业PM软件 | ❌ 初期不做 |
| 自定义报表/BI | 管理驾驶舱固定维度足够 | ❌ 后期再说 |
