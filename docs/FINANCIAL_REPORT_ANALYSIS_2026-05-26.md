## 二、逐表深度分析

### 1️⃣ 月度/年度报表

**逻辑评价：** ✅ 基础逻辑正确，但过于简单——仅仅是收入减支出的差额，不是真正的损益表。

**数据来源问题：**
- `Income.amount` 包含银行流水导入的所有收入（含内部公司转账），未被排除
- `Expense.amount` 包含银行流水导入的所有支出（含工资、社保、税费等），未做分类
- 收入/支出在同一个 `date`/`expense_date` 下的简单加总，**没有任何分类维度**

**建议改进点：**
- 收入应分科目（主营业务收入/其他业务收入/营业外收入）
- 支出应分科目（营业成本/管理费用/销售费用/财务费用）
- 排除内部公司之间的转账（交叉持股公司间资金调拨不应算收入/支出）

### 2️⃣ 工资汇总

**逻辑评价：** ✅【已修复】社保公司部分已从 SocialRecord 精准读取，不再用反推公式。

**数据来源：** WageRecord（工资条级数据）
- `gross_salary` 应发合计 ✓
- `net_salary` 实发合计 ✓
- `tax` 个税 ✓
- `social_insurance` 个人社保扣款 ✓
- `housing_fund` 个人公积金扣款（已改为从 SocialRecord 精准读取）✓
- SocialRecord：公司社保（6项单位合计）✓、公司公积金 ✓

**问题：**
- 59条 WageRecord = 59人次工资，但实际社保只有4条 SocialRecord（仅1月导入）
- 社保数据补全后才能准确显示公司社保/公积金成本

### 3️⃣ 发票汇总

**逻辑评价：** ⚠️ 基本逻辑正确，但缺少关键维度

**问题：**
- 没有区分发票类型（专票 vs 普票）的汇总
- 没有按客户/供应商维度统计
- 缺少 `tax_amount` 的完整汇总（目前只汇总了 amount，tax_amount 在summary中正确但results中没有逐公司统计）

### 4️⃣ 现金流量表

**逻辑评价：** ✅ 从 BankStatement 读取，余额计算逻辑已修复（优先取银行快照余额），比依靠收入-支出推算更准确。

**问题：**
- **不是真正的现金流量表！** 标准现金流量表按**经营活动/投资活动/筹资活动**三大类分类，而当前的"现金流量表"仅仅是银行账户余额变动表
- 没有对银行流水进行"活动类型"分类
- 期初余额从 BankStatement.balance 取，但只取最近一条记录，如果跨年数据不全可能不准

### 5️⃣ 应收应付账龄分析

**逻辑评价：** ⚠️ 账龄分桶逻辑正确，但数据源有问题

**问题：**
- 应收（AR）= Invoice(type='income', status='pending')——但发票有"待收款/已付款/已作废"状态，**仅"待收款"状态**的才是应收
- 实际上 Invoice 模型 `STATUS_CHOICES = [('pending', '待收款/待付款'), ('paid', '已完成'), ('cancelled', '已作废')]`
- 应收账龄应该基于 **due_date（到期日）** 而非 issue_date（开票日期），当前代码是取 `due_date or date or today`，若业务未填写due_date则退化为开票日期，账龄失真
- 559条发票中，有多少有 due_date 需要实际检查

### 6️⃣ 客户收入排行

**逻辑评价：** ✅ 排除了内部公司转账（从 Company 表动态读取公司名单），排名逻辑清晰

**问题：**
- 按客户名称字符串分组，同样的客户如果写法不同（如"南方科技大学"和"南方科大"）会被分为两个客户
- 没有关联到 CRM Client 表做标准化（CRM里有正式客户名）
- 当前有 219 条 Income，但 CRM 客户数据可能没有完整覆盖

### 7️⃣ 供应商支出

**逻辑评价：** ✅【已修复】区分企业供应商/个人供应商/无供应商三类，从 CRM Supplier 匹配真实供应商

**问题：**
- 同客户排行一样的问题：供应商名称字符串匹配
- 个人供应商识别用正则 `^[\u4e00-\u9fa5]{2,4}$`（纯中文2-4字），但实际公司名也可能只有2-4字，存在误判风险
- 975条 Expense 中，需要确认供应商名称的覆盖率

### 8️⃣ 税费汇总

**逻辑评价：** ⚠️ 逻辑已修复多次（P1-1~P3-1），但仍然存在风险

**数据来源：**
- Invoice.tax_amount（进项/销项税）
- Expense(expense_category='税款') + 描述解析（根据税单号推断税种）

**问题：**
- **税费数据源不完整**：实际缴纳的税款需要同时从银行流水+手工录入两个渠道获取，但目前只从 Expense 表取
- **税单号解析脆弱**：`_parse_tax_type()` 依赖于描述中的"税单号:XXXX"格式，如果银行推送的描述格式变化，解析会失败
- **社保反推在税费汇总中仍然存在**：行525-557，当有 CompanySocialConfig 时仍然用反推公式计算社保，只有无配置时才走 SocialRecord。两者逻辑不一致
- 没有区分 **增值税销项 vs 进项抵扣**（应交增值税 = 销项 - 进项），当前仅做了加总
- 工资个税 = WageRecord.tax 未在此处计算（目前只从银行实际缴税记录取），如果个税数据从WageRecord来更准确

### 9️⃣ 预算执行表

**逻辑评价：** ⚠️【已修复P0-1】expense_type → expense_category 关键词匹配

**问题：**
- **没有预算数**（预算金额），只有实际支出——所以叫"预算执行表"但没有"预算"只有"执行"
- 关键词匹配 `expense_category__icontains='办公'` 匹配到"办公费用"但也会匹配"办公室用品"等
- 工资从 WageRecord 读取（正确），社保在无 config 时从 SocialRecord 读取（正确），但部分重复计算风险：工资中的社保部分 + 预算的"社保公积金"行 = 社保被算两次
- 类型"other"没有匹配关键词，所有无匹配的支出都被吞掉了

### 🔟 资产负债表（balance_sheet）

**逻辑评价：** ❌ **名不副实，这是当前最严重的问题**

- 名为"资产负债表"，实际**仅仅是收入减支出的收支差额表**
- **真正的资产负债表**公式：资产 = 负债 + 所有者权益
- 当前系统的数据模型**没有任何资产、负债、权益科目**
- 缺少：
  - 货币资金（可从 BankStatement 余额推算，但需汇总多银行账户）
  - 应收账款（可从 Invoice 推算，但仅限发票业务）
  - 固定资产（无模型）
  - 应付账款（可从 Invoice 推算）
  - 实收资本/未分配利润（无模型）
  - 预收/预付款（无跟踪）

## 四、与标准财务报表体系对比

### 企业三大主表

| 标准报表 | 当前系统 | 差距 |
|---------|---------|------|
| **资产负债表**（Balance Sheet） | ❌ 不存在 | 需新建整个模型：资产（货币资金/应收/存货/固资）、负债（应付/预收/借款）、权益（实收资本/未分配利润） |
| **利润表**（Income Statement） | ❌ 不存在 | 需新建：营业收入→营业成本→毛利→期间费用→营业利润→利润总额→净利润 |
| **现金流量表**（Cash Flow Statement） | ⚠️ 仅有余额变动表 | 需对银行流水按活动分类：经营/投资/筹资 |

### 辅助报表

| 标准报表 | 当前系统 | 差距 |
|---------|---------|------|
| **应收账龄分析** | ⚠️ 有，但不完整 | 缺少预付款/其他应收款分析 |
| **应付账龄分析** | ⚠️ 有，但不完整 | 缺少预收款/其他应付款分析 |
| **费用明细表** | ⚠️ 有（预算执行表） | 缺预算数 |
| **税务申报表** | ✅ 税费汇总 | 基本覆盖，但逻辑需统一 |
| **工资汇总表** | ✅ | 5月已修复，社保数据依赖 SocialRecord 补全 |
| **收入明细表** | ⚠️ 有月度报表 | 没有分产品/分项目收入分析 |

### 关键缺失报表

| 缺失报表 | 说明 | 优先级 |
|---------|------|-------|
| 利润表（P&L） | 企业最核心报表之一，管理层最关注 | **P0** |

## 五、数据模型缺失分析

### 现有模型能支撑的报表

```
Income      → 收入分析、客户排行、月度收支
Expense     → 支出分析、供应商排行、预算执行
Invoice     → 发票汇总、应收应付账龄、税费（税额）
WageRecord  → 工资汇总、个税（部分）
BankStatement → 余额变动、现金收支
SocialRecord → 社保成本、工资汇总（公司部分）
```

### 要实现标准财务报表需要的模型扩展

```
📌 会计科目表 (Chart of Accounts)
   ├── 资产类：货币资金、应收票据、应收账款、预付账款、其他应收款、存货、固定资产、累计折旧
   ├── 负债类：应付票据、应付账款、预收账款、应付职工薪酬、应交税费、其他应付款、长期借款
   └── 权益类：实收资本、资本公积、盈余公积、未分配利润

📌 科目余额表 (Account Balance) — 按月/年
   ├── 期初余额、本期发生(借方/贷方)、期末余额
   └── 支撑试算平衡和三大主表编制

📌 辅助核算维度
   ├── 部门维度 → 部门费用分析
   ├── 项目维度 → 项目损益分析（已有 Project 模型，但未做损益归集）
   └── 客户/供应商维度 → 已有 CRM，但未做标准化
```

## 六、专业建议分级

基于当前系统定位（多公司经营聚合系统，非会计软件），建议分阶段优化：

### 🔴 P0 阶段（尽快修复）

1. **改表名** — `balance_sheet` → `revenue_expense_summary`，"资产负债表" → "收支汇总表"
2. **改表名** — `现金余额变动表`，"现金流量表" → "银行余额变动表"
3. **税费汇总统一社保来源** — 全部走 SocialRecord，不再用反推公式

### 🟡 P1 阶段（架构优化）

4. **增值税计算修复** — 应交增值税 = 销项 - 进项
5. **多公司内部转账排除** — 统一在所有报表中系统性地排除
6. **科目化** — 引入二级分类（收入/支出各有2-3级科目）

### 🟢 P2 阶段（功能增强）

7. **CRM 客户/供应商标准化** — 收入/支出记录关联到 CRM 实体
8. **发票多维度汇总** — 按客户/供应商/类型/税率
9. **预算数录入界面** — 让预算执行表名副其实
10. **数据口径文档** — 建立每个报表的数据来源、计算逻辑、业务含义文档

### 🔵 P3 阶段（专业化）

11. **会计科目表模型** — 引入 CoA 和科目余额表
12. **利润表** — 基于科目余额表编制
13. **资产负债表** — 基于科目余额表编制
14. **数据库校验规则** — 借贷平衡检查
