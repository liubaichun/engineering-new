## 一、群通知 vs 私信通知

### 群通知

| 项目 | 说明 |
|:----|:-----|
| **定义** | 一条消息发到群里，群内所有人都能看到 |
| **典型场景** | 系统公告、月结提醒、公司通告 |
| **谁配** | 管理员 |
| **配置一次，永久使用** | ✅ |

**配置步骤（管理员）：**
```
1. 在飞书/钉钉/企微创建一个群机器人
2. 复制 Webhook URL
3. 进系统后台 → 通知渠道 → 添加渠道
4. 选渠道类型（飞书/钉钉/企微）→ 用途选「群通知」
5. 粘贴 Webhook URL → 保存
6. 系统自动发测试消息到群里 → 确认收到
```

**发送逻辑：**
```
业务事件 → 查该公司的群通知渠道有哪些
         → 逐个 POST 到 Webhook URL
         → 群里所有成员看到
```

**不需要用户参与**，管理员配好就行。

---

### 私信通知

| 项目 | 说明 |
|:----|:-----|
| **定义** | 一条消息发到指定个人的 IM/邮箱，只有他收到 |
| **典型场景** | 审批待办推给审批人、任务指派推给负责人、合同到期推给合同负责人 |
| **谁配** | 管理员配渠道 + 用户自己绑定 |
| **用户只需绑定一次** | ✅ |

**配置分两步：**

**第1步：管理员配渠道（配一次就行）**
```
管理员配好私信渠道：
  飞书私信渠道 → 填 App ID + App Secret（用于调用飞书会话 API）
  钉钉私信渠道 → 填 AppKey + AppSecret + AgentId
  微信私信渠道 → 填 PushPlus 账号信息（作为中转）
  邮箱渠道     → 填 SMTP 主机 + 端口 + 账号
```

**第2步：用户绑定自己的账户（每个用户绑一次）**

## 二、三种绑定形式（关键是让用户省事）

### 形式一：扫码绑定（最佳体验）

适用于：**飞书、钉钉、企业微信**

```
用户在系统里点「绑定飞书」
        ↓
系统生成一个二维码（OAuth 授权链接）
        ↓
用户打开飞书扫码
        ↓
飞书弹出授权页面 → 用户点「确认」
        ↓
系统自动拿到用户的 open_id
        ↓
绑定完成 ✅ 用户零输入
```

**用户操作量：** 点 1 次按钮 + 扫 1 次码 + 点 1 次确认 = **3 步**
**代码现状：** `FeishuPlugin`、`DingtalkPlugin`、`WecomPlugin` 都已实现这套流程
**二维码管理页面：** 已有 `BindingQRCodeView` 生成二维码 URL

---

### 形式二：手动填 Token（备选方案）

适用于：**微信（通过 PushPlus）**

```
用户在系统里点「绑定微信」
        ↓
看到操作说明：
  第1步：微信扫码关注 PushPlus 公众号
  第2步：在公众号里回复「token」获取你的专属 Token
  第3步：把 Token 复制粘贴到下方输入框
        ↓
用户填完 → 点「绑定」
        ↓
系统验证 Token 有效 → 绑定完成
```

**用户操作量：** 关注公众号 + 复制 Token + 粘贴 + 点绑定 = **4 步**
比扫码麻烦一点，但微信没有个人 OAuth API，这是最实用的方案。

---

### 形式三：自动匹配（最省事）

适用于：**邮箱**

```
无需任何绑定操作。
员工在「员工档案」里填了邮箱 → 系统自动用它发邮件通知。
```

**用户操作量：** 0 步（HR 录入员工信息时填好就行）

## 五、发送逻辑完整流程

### 群通知

```
业务事件触发（如系统管理员发了公告）
        ↓
确定公司 ID
        ↓
查 ChannelPlugin：company_id=当前公司, usage='broadcast', is_active=True
        ↓
遍历找到的群渠道，逐个发送：
  飞书群 → POST 飞书群机器人 Webhook
  钉钉群 → POST 钉钉群机器人 Webhook
        ↓
记录 NotificationLog（每个渠道一条）
```

### 私信通知

```
业务事件触发（如张三提交审批，指定李四审批）
        ↓
确定接收人：李四（User 对象）
        ↓
确定公司 ID
        ↓
查 ChannelPlugin：company_id=当前公司, usage='personal', is_active=True
        ↓
查 UserNotificationPreference：李四是否开启了该事件类型
  关闭了 → 跳过
  开启了 → 继续
        ↓
查 ChannelBinding：user=李四, channel__in=[上面的渠道列表], is_active=True
        ↓
遍历找到的绑定，逐个发送：
  李四绑了飞书（open_id=ou_xxx）→ FeishuPlugin.send_message(ou_xxx, ...)
  李四绑了微信（token=psh_xxx）→ WechatPlugin.send_message(psh_xxx, ...)
        ↓
记录 NotificationLog（每个绑定一条）
```

## 七、实施路径（简化版）

### 第1步：加 usage 字段 + 清洗数据

- `ChannelPlugin` 加 `usage` 字段：`broadcast` / `personal`
- 数据迁移：有 webhook_url 的 → broadcast；有 app_id/app_key 的 → personal

### 第2步：修通道连通性

- 飞书：重刷 token → 测试发送
- 微信：确认 PushPlus token 能正常推送
- 邮件：后台加 SMTP 配置项 → 测试发送

### 第3步：补用户绑定的前端页面

- 把二维码扫码绑定做进用户前端页面（当前只有 admin 的绑定管理，没有用户的扫码页面）
- 补微信绑定的说明和表单

### 第4步：接入业务事件（先接审批）

- 审批创建 → 推给审批人
- 审批结果 → 推给申请人
