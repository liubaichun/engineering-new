# 通知系统优化方案 v5.0

> **版本：** v5.0  
> **日期：** 2026-05-28  

---

## 一、设计原则

```
1. 每家公司自己配渠道——用什么不用什么，自己定
2. 微信能发到个人——不是群发，是点对点推给张三的微信
3. 邮件留配置入口——用户配好 SMTP 就能用
4. 不搞复杂路由——渠道配好了，业务事件直接走配置发
```

---

## 二、支持哪些渠道

| 渠道 | 发送方式 | 能发到哪 | 现状 |
|:----|:---------|:---------|:-----|
| **飞书** | 群机器人 Webhook | 飞书群（所有人都看到） | ✅ 代码已有 |
| **飞书（个人）** | 会话 API（open_id） | 指定用户的飞书个人消息 | ✅ 代码已有 |
| **钉钉（个人）** | 应用消息 API | 指定用户的钉钉个人消息 | ✅ 代码已有 |
| **企业微信（个人）** | 应用消息 API | 指定用户的企业微信消息 | ✅ 代码已有 |
| **微信（个人）** | PushPlus API | 用户个人微信（关注公众号后） | ✅ 代码已有 |
| **邮件** | SMTP | 指定邮箱 | ⚠️ 缺后台配置页面 |

**关键说明——微信怎么发到个人：**

```
用户在公司系统里点「绑定微信」
  → 填自己在 PushPlus 拿到的 Token
  → 系统存下 Token
  → 以后发通知时：POST 到 PushPlus API
  → PushPlus 公众号给这个用户发消息
  → 用户微信里收到

这个 Token 是个人唯一的，所以只发给你，不是群发。
```

每家公司：
- 各自独立——百川配飞书不影响龙晟

- 配置入口已在代码中（`ChannelPlugin` 模型绑定 `Company`）

### 3.2 添加一个渠道时选什么

```
添加渠道时，管理员选：
  1. 渠道类型：飞书 / 钉钉 / 企微 / 微信 / 邮件
  2. 渠道用途：群通知（发到群里） / 个人通知（发到个人）
  3. 填写配置：
     - 飞书群 → Webhook URL
     - 飞书个人 → App ID + App Secret  
     - 钉钉个人 → AppKey + AppSecret + AgentId
     - 微信个人 → PushPlus Token
     - 邮件 → SMTP 主机 + 端口 + 账号 + 密码
  4. 保存 → 系统自动测试连通性
```

## 四、通知怎么发

### 4.1 群通知（群机器人）

```
管理员配好群机器人渠道（飞书群 Webhook / 企微群 Webhook 等）
        ↓
业务事件（如系统公告、月结提醒）
        ↓
系统查：这个公司的「群通知渠道」有哪些？
        ↓
逐一发到这些群机器人的 Webhook
        ↓
群里所有人都收到
```

**只发一次，群里所有人都看到。** 不需要知道谁是谁。

### 4.2 个人通知（点对点）

```
用户通过以下方式绑定自己的账户：
  - 飞书：扫码授权（自动获取 open_id）
  - 微信：填 PushPlus Token（系统调用 PushPlus API）
  - 没有绑定 → 不推送

业务事件（如审批待办、任务指派、合同到期）
        ↓
系统查：这个事件相关的接收人是谁？
  - 审批待办 → 当前审批节点的人
  - 任务指派 → 任务的 assignee
  - 合同到期 → 合同 created_by
        ↓
对每个接收人：
  查这个用户的个人渠道绑定
  查这个公司有哪些个人渠道
  取交集 → 逐渠道发送
        ↓
微信收到 / 飞书收到（个人消息） / 钉钉收到 / 邮件收到
```

### 4.3 发送逻辑一句话概括

```
群通知 → 查公司有哪些群渠道 → 逐个发
个人通知 → 查接收人有哪些绑定 → 逐个发
```

**不需要额外的映射表，不需要复杂的路由规则。** 渠道本身就是配置。

## 五、当前系统的差距

| 需求 | 现状 | 要改什么 |
|:----|:-----|:---------|
| 每家公司独立配渠道 | ✅ `ChannelPlugin` 已绑定 `Company` | 无改动 |
| 渠道分群通知/个人通知 | ❌ 现有渠道没分 | `ChannelPlugin` 加 `usage` 字段 |
| 微信个人推送 | ✅ `WechatPlugin` 已有（PushPlus） | 验证连通性 |
| 飞书个人推送 | ✅ `FeishuPlugin` 已有（open_id） | Token 过期，需修复 |
| 邮件 SMTP 可后台配 | ❌ 只读环境变量 | 加 SMTP 配置页面 |
| 业务事件触发通知 | ❌ 没接入 | 每个事件 view 加一行代码 |

## 六、实施步骤

### 第一步：渠道加 usage 字段 + 页面调整

- `ChannelPlugin` 加 `usage` 字段（`broadcast` / `personal`）
- 渠道管理页面的添加/编辑弹窗加用途选择
- 已有渠道做数据迁移：有 webhook_url 的标 broadcast，有 app_id 的标 personal

### 第二步：修通道

- 飞书：重刷 token，点「测试发送」验证
- 微信：添加一个微信渠道（PushPlus），测试发送
- 邮件：在系统设置里加 SMTP 配置输入框和测试按钮

### 第三步：接入业务事件

选最重要的先接：

```
第一波（审批通知）：
  审批创建  → 通知当前审批人
  审批结果  → 通知申请人

第二波（合同/任务通知）：
  合同到期预警  → 通知负责人
  任务指派      → 通知负责人
```

每接入一个，就在对应 view 的 create/update 末尾加一行：

```python
ChannelNotificationService.send(
    user=approver,
    title='【待审批】费用报销申请',
    content='张三提交了一笔 ¥3,500 的费用报销',
    notification_type='approval',
)
```

### 第四步：验证

- 百川公司配飞书群 + 飞书个人 + 微信个人
- 创建一个审批 → 审批人收到飞书个人消息 + 微信消息
- 发一个群通知 → 飞书群收到
- 通知日志能看到记录
